# Data Vertical Review — 2026-06-04

Trigger: `workflow_dispatch` (full audit). Branch: `master` @ `535df15`.
Scope: routing config schema, cost-tracker model coverage, session
manager backwards compatibility, router fallback safety, and the
documentation surface that pairs with these schemas.

## Summary

The data surface is internally consistent for the live `routing-config.json`
shipped on `master`. All six model ids referenced by routing have matching
`MODEL_PRICING` entries, both JSON files parse cleanly, and the router
honours the documented decision order (force -> rule -> score).

Two material risks need attention before they ship into a release:

1. **Session schema has no `version` field and no migration code.** The
   role definition mandates versioned migration for `~/.alexi/sessions/`,
   but `Session` / `SessionMetadata` evolves silently (the `workdir`
   field was added without one). A future renaming or type narrowing
   will silently corrupt user data.
2. **`routing-config.example.json` is drifting from `routing-config.json`.**
   It still advertises retired model ids (`claude-3.5-sonnet`,
   `claude-4-sonnet`) and is missing the `enabled` flag, the haiku
   variant, and the opus variant. New users copying the example get
   models that do not exist in the cost tracker.

Severity legend: `block` (must fix before merge / release), `warn`
(should fix soon), `info` (track but no immediate action).

## Session compatibility

| Severity | Finding |
| --- | --- |
| `block` | `Session` / `SessionMetadata` (`src/core/sessionManager.ts:23-52`) has no `version` field. Per the data role mandate ("introduce a `version` field in any schema you own"), every persisted schema must carry one. The `workdir` field was already added as optional with a defensive read path — that is the right pattern but it does not scale: future renames of `tokens.input` or `messages[].role` will not be caught by tests. Add `metadata.version: 1` on write, treat missing as `version 0`, and gate any future change on incrementing it. |
| `warn` | `loadSession` (`src/core/sessionManager.ts:106-123`) does `JSON.parse(content) as Session` with no shape validation. Hand-edited or partially-written session files will be loaded as `Session` and crash the orchestrator on the first access to `session.messages`. Wrap with a Zod schema (or a minimal shape guard: `Array.isArray(parsed.messages) && parsed.metadata?.id`). |
| `info` | The optional `workdir` field is correctly handled by `listSessions` (`src/core/sessionManager.ts:232-240`) — legacy sessions without a workdir are filtered out when a workdir filter is active. This is the model the rest of the schema should follow. |
| `warn` | `Message.tokens` is `{ input?: number; output?: number }` and `addMessage` (`src/core/sessionManager.ts:157-159`) reads them with `|| 0`. If a future change tightens the type to required, every legacy session message will break. Keep both fields optional permanently and document that contract. |

## Router safety

| Severity | Finding |
| --- | --- |
| `block` | `routePrompt` (`src/core/router.ts:226-242`) calls `scored[0]` and then `best.score`. If `getModelRegistry()` returns an empty array (every model disabled, or `availableModels` filter eliminates everything), `best` is `undefined` and the function throws `TypeError: Cannot read properties of undefined`. Guard with: if `scored.length === 0`, return a `RoutingDecision` pointing at `config.preferences.fallbackModel` with `confidence: 0` and `reason: 'No candidates matched; fell back to default'`. Tests under `tests/router*.test.ts` should cover the empty-candidates case. |
| `warn` | `config.preferences.fallbackModel` is declared in `RoutingPreferences` (`src/config/routingConfig.ts:25-30`) but never actually read by `routePrompt`. The role mandate "fallback chains never end without a default model" is not enforced in code today — it is documented intent only. Wire it into the empty-candidates path above. |
| `info` | The forced-model branch (`src/core/router.ts:175-182`) and the rule-with-modelId branch (`src/core/router.ts:190-199`) both return well-formed `RoutingDecision`s. The score-based path is the only one with the bug. |
| `info` | `explainRouting` (`src/core/router.ts:248-304`) shares `getModelRegistry` and the same scoring logic, so any registry-empty fix benefits both paths. |

## Recommendations

In priority order:

1. **(block)** Add an empty-candidates guard to `routePrompt` and a regression test under `tests/router.test.ts` that disables every model and asserts the result targets `preferences.fallbackModel` with a sensible `reason`. Wire `preferences.fallbackModel` into the path so the JSON contract becomes load-bearing.
2. **(block)** Introduce `metadata.version` on `SessionMetadata`. Bump the writer to emit `1`, treat absent as `0`, and add a migration test that reads a v0 fixture (no `workdir`, no `version`) and writes a v1 file losslessly. Place the fixture under `tests/fixtures/sessions/v0-session.json`.
3. **(warn)** Add `MODEL_PRICING` entries for `anthropic--claude-4.7-opus` (and any `sap-ai-core/`-prefixed aliases the agent fleet uses). Reference the SAP AI Core deployment URL in the PR description. Same PR: prune `gpt-4.1-mini` / `gpt-4.1-nano` from `MODEL_PRICING` OR add them to `routing-config.json`.
4. **(warn)** Realign `routing-config.example.json` with the live shape: drop `claude-3.5-sonnet` / `claude-4-sonnet`, add the haiku and opus variants, and include `"enabled": true` on each model. Same PR: replace the `docs/ROUTING.md` 5-line stub with the actual schema reference (rules, conditions, preferences, model fields including `enabled`). The `documentation-update.yml` workflow currently overwrites this file — coordinate with the documentation-agent owner so the regenerated content includes this material instead of being clobbered.
5. **(warn)** Promote `enabled?: boolean` into `ModelCapability` (`src/core/router.ts:12-19`) so the runtime filter at `getModelRegistry` is type-safe, then remove the inline cast.
6. **(warn)** Add a Zod parse to `loadSession` and `loadRoutingConfig`. Both currently `JSON.parse + as T`; both fail catastrophically on malformed user data.

None of these are speculative — every item maps to a concrete file and line above. Items 1 and 2 are the real risks; the rest are cleanup that makes the schema layer honest about what it owns.
