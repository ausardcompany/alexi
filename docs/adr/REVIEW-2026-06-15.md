## Summary

Third architecture run. Two prior runs (`docs/adr/REVIEW-2026-06-04.md`,
`docs/adr/REVIEW-2026-06-04T12-followup.md`) introduced ADR 001 and
catalogued findings F1..F9 against the provider surface, layering, and
documentation drift. Between the previous run and this one, eleven days

on 2026-06-06, 06-07, 06-08, 06-09, 06-10, 06-11, 06-12, 06-13, 06-14)
plus several factory and dependency commits.

All structural findings F1..F8 from the earlier runs remain open and
unchanged. F3b (provider-leak in `streamingOrchestrator.ts`) is also
unchanged. **Three new structural issues have appeared via the upstream

- **F10 `block`** -- `src/core/billing.ts` was added by upstream sync. Its
  contents are Stripe-style `payment_method_options` boilerplate that has
  no relationship to Alexi's domain (LLM orchestration). It is not
  imported by any other file. This is unambiguous foreign code that
  should not be in `src/core/`.
- **F11 `block`** -- `src/core/catalog.ts` was added by upstream sync. It
  re-defines a Zod-schema-backed type named `ModelCapability` -- the
  exact same name as the `interface ModelCapability` already exported
  from `src/core/router.ts:12` and consumed by `src/config/routingConfig.ts`.
  Two different types with the same name in the same `src/core/`
  package is a name-collision land-mine. The new file is not imported
  by anything yet.
- **F12 `warn`** -- `src/core/integration/` is a brand-new top-level
  subdirectory under `src/core/` containing only `schema.ts`, whose
  full content is a single comment line:
  `// Updated schema code reflecting new credential structure`.
  No code, no exports, no consumers. Either dead artefact from an
  upstream merge or an in-progress feature that landed without its
  payload.

F6 has also grown: `src/core/migration/` now contains five SQL artefacts
(was one) including two named subdirectories that look like Prisma
migration folders. There is still no SQLite / Prisma dependency in
`package.json`, so the directory remains entirely unwired -- but it has
gained mass since the last review.

**No new ADR is created in this run.** As with the 2026-06-04T12
follow-up, the rules these findings violate are already covered by
ADR 001. The right next step is the bulk backfill ADR (002) plus
deletion / quarantine of F10/F11/F12, not more rule documents.

### F10 -- `block` -- Stripe-shaped foreign code in `src/core/billing.ts` (NEW)

`src/core/billing.ts` (10 lines, full content):

```ts
export const paymentOptions = {
  payment_method_options: {
    card: {
      setup_future_usage: 'off_session',
    },
    link: {
      setup_future_usage: 'off_session',
    },
  },
};
```

This is verbatim Stripe `PaymentIntent.payment_method_options` shape.
Alexi has no payment domain, no Stripe dependency, and no caller of
`paymentOptions` (`grep -rln "paymentOptions" src/` returns nothing).

The file appeared in the 2026-06-11 / 2026-06-12 upstream sync stream
(commits `1b0df735`, `adf6f15a`). Likely the upstream merge tooling
copied it from a different repository without filtering. Constitution I
states: "Provider-Native LLM Orchestration. The system runs entirely
through SAP AI Core ...". A `core/billing.ts` with credit-card logic is
out of scope on every dimension: domain, dependency, layering. It must
be removed.

### F11 -- `block` -- `ModelCapability` name collision in `src/core/catalog.ts` (NEW)

`src/core/catalog.ts:7` defines:

```ts
export const ModelCapability = z.enum([
  'text', 'vision', 'function_calling', 'json_mode', 'streaming',
]);
// eslint-disable-next-line no-redeclare
export type ModelCapability = z.infer<typeof ModelCapability>;
```

`src/core/router.ts:12` already exports a different type with the same
name:

```ts
export interface ModelCapability {
  id: string;
  type: 'openai' | 'claude' | 'gemini';
  costTier: 'cheap' | 'medium' | 'expensive';
  // ...
}
```

`src/config/routingConfig.ts:8` consumes `router.ts`'s definition:

```ts
import type { ModelCapability } from '../core/router.js';
```

The catalog file is currently dead (`grep -rln "core/catalog" src/`
returns nothing), which is the only reason this has not blown up. The
moment any caller imports `from '../core/catalog.js'`, the next reader
will read the wrong type without a TypeScript error and lose hours.
The `eslint-disable no-redeclare` smell is a strong signal that this
file was lifted from a different codebase that does not have the
`router.ts` definition.

Two acceptable resolutions: (a) delete `src/core/catalog.ts` until
there is a real driver requiring a model-capability registry distinct
from `router.ts`; (b) rename the new type (e.g. `CatalogCapability` or
`ModelFeature`) and write an ADR explaining how the two registries
relate. Option (a) is preferred.

In all three cases the right action is to delete the file (and the now

a tracking issue with the auto-implement label per the AGENTS.md
escalation pattern. Per ADR 001 the addition of a new top-level src
sub-directory should have been accompanied by an ADR; this one was not.

## Recommendations

1. **Combined cleanup PR -- `chore(sync)` or `refactor(core)`** -- delete
   `src/core/billing.ts`, `src/core/catalog.ts`, and
   `src/core/integration/schema.ts` (and the empty `integration/`
   dir). Verify with `npm run typecheck && npm run lint && npm run build`
   that nothing referenced them. This unblocks F10, F11, F12 in one
   step.

2. **Provider-surface PR -- `refactor(providers)`** (carried forward from
   the previous run) -- still the right next architectural fix:
   - Re-export `type CompletionResult, type TokenUsage` from
     `src/providers/index.ts` (type-only, zero runtime cost).
   - Re-export `buildSessionHeaders` from `src/providers/index.ts`.
   - Add a `listDeployments()` helper to `src/providers/index.ts`
     wrapping `DeploymentApi.deploymentQuery(...)`.
   - Update three callers: `src/cli/commands/models.ts:6+87`,
     `src/core/agenticChat.ts:19`, `src/core/streamingOrchestrator.ts:16`.

3. **Stop the bleed in upstream sync.** F10 / F11 / F12 were all
   introduced by `feat(sync): apply upstream changes` PRs in the past
   week. The two-stage planning + execution sync described in
   `AGENTS.md` is meant to filter foreign domain code, but it clearly
   is not catching Stripe boilerplate or empty placeholder files. A
   minimal mitigation that the Quality / Infrastructure verticals can
   add without architectural change: run `git diff` over the synced
   tree against an explicit allowlist of expected paths, and refuse to
   merge if the diff introduces files matching `payment*`, `billing*`,
   `*credentials*`, or new top-level `src/<x>/` directories without an
   ADR file in the same PR.

4. **Delete the Kotlin files (F5).** Two reviews in a row have flagged
   `src/permission/PermissionView.kt` and `PermissionViewTest.kt`.
   They cannot run, cannot be tested, and cannot be type-checked in
   this build. A one-line `git rm` PR scoped `chore(permission)` would
   clear it.

5. **F6 -- migration directory decision.** With the SQL footprint having
   grown five-fold since the previous review, decide now: either ADR +
   `better-sqlite3` dep + loader + tests, or `git rm -r src/core/migration/`.
   The current state (Prisma-style migration folders, no Prisma) is the
   worst of both worlds.

6. **F7 -- update role-architecture.md module map.** The role-prompt
   tells future agents the module map is ~10 modules. Reality is 32.
   Either replace the inline module map with a pointer
   ("`docs/ARCHITECTURE.md` is the canonical view") or refresh the list.

## ADR needed?

**No new ADR in this run.** The structural rules ADR 001 already
codifies are sufficient to cover F10..F12 -- they are violations of
those rules, not new categories. Filing a separate "ADR 002 -- Foreign
code in core" would dilute the audit trail.

What this run adds is one more data point that the **bulk backfill ADR
(ADR 002)** is overdue. With F10..F12 in the tree, the moment ADR 002
is written it will need to either (a) explicitly call out the orphan
files for removal as part of the baseline cleanup, or (b) memorialise
them as accepted, which is unacceptable. Resolving the deletions in
Recommendation 1 first means ADR 002 can describe a clean baseline.

The next ADR worth writing remains:

- **ADR 002 -- Backfill: accepted-as-baseline `src/` top-level layout**,
  to be filed *after* the cleanup PRs (Recommendations 1, 2, 4, 5)
  land. ADR 002 should also formalise the "tests under `__tests__/`
  and `tests/` may import across layers" exception (resolves F9).

The ADR 003..005 follow-ups listed in ADR 001 still apply unchanged.
