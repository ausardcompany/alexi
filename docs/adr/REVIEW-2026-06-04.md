# Architecture Review — 2026-06-04 (scheduled, master)

## Summary

The `src/` tree has grown from the documented ~10 top-level modules to **32**
top-level directories, none of which were accompanied by an Architecture
Decision Record (ADR). `docs/adr/` did not exist on disk. ESM `.js` import
discipline is clean (the only "bare" hits are inside test template strings).
The provider abstraction (constitution III) is mostly intact at the *core*
layer, but two real leaks exist: `src/cli/commands/models.ts` reaches into
`@sap-ai-sdk/ai-api` directly, and `src/core/agenticChat.ts` imports a type
straight from `src/providers/sapOrchestration.ts` instead of going via
`providers/index.ts`. There is also a bidirectional import between
`src/tool/tools/*` and `src/agent/agentsMdReminders.ts` which is a real
layering inversion, plus stray `.kt` files and an orphan `.sql` migration.

This run created `docs/adr/` and added **ADR 001 — Establish ADR Process and
Layering Rules** to make the rules explicit and auditable. `docs/ARCHITECTURE.md`
was updated to reflect the actual current top-level layout (32 dirs, not 18).

## Findings

### F1 — `block` — No ADRs on file despite extensive structural change

Seventeen new top-level `src/` directories were merged since the original
architecture doc was written: `command/`, `compaction/`, `context/`,
`doctor/`, `flag/`, `git/`, `i18n/`, `init/`, `log/`, `plan/`, `profile/`,
`reference/`, `share/`, `sound/`, `sync/`, `undo/`, `update/`. The
constitution and `AGENTS.md` both name ADRs as the gate for non-trivial
structural change. `docs/adr/` did not exist until this run.

### F2 — `block` — Provider SDK leak in CLI layer (constitution III)

`src/cli/commands/models.ts:6` imports `@sap-ai-sdk/ai-api` directly:

```ts
import { DeploymentApi } from '@sap-ai-sdk/ai-api';
```

and calls `DeploymentApi.deploymentQuery(...)` at line 87. The rule "rest
of codebase imports the provider interface only; concrete SDK calls live
exclusively in `src/providers/*.ts`" (role-architecture.md) is violated.

### F3 — `warn` — Direct concrete-provider import in core

`src/core/agenticChat.ts:19`:

```ts
import type { CompletionResult, TokenUsage } from '../providers/sapOrchestration.js';
```

These types should be re-exported from `src/providers/index.ts` so callers
in `src/core/` can stay agnostic of which concrete file implements the
provider. Test files (`__tests__/aliases.test.ts`,
`__tests__/agenticChat.test.ts`) have the same issue but tests get more
slack.

### F4 — `warn` — Layering inversion: `src/tool/` -> `src/agent/`

`src/tool/tools/{read,write,edit,multiedit,grep,task}.ts` import from
`src/agent/`:

| File | Import |
|------|--------|
| `src/tool/tools/grep.ts:19` | `attachAgentsMdRemindersForPaths` from `../../agent/agentsMdReminders.js` |
| `src/tool/tools/edit.ts:9` | `attachAgentsMdReminders` from `../../agent/agentsMdReminders.js` |
| `src/tool/tools/multiedit.ts:9` | same |
| `src/tool/tools/read.ts:18` | same |
| `src/tool/tools/write.ts:11` | same |
| `src/tool/tools/task.ts:7` | `getAgentRegistry, type Agent` from `../../agent/index.js` |

Meanwhile `src/agent/agentsMdReminders.ts:11` imports `type { ToolResult }`
from `../tool/index.js`. The type-only direction is fine, but the
runtime-value direction (tool importing agent) inverts the documented
layering. The `agentsMdReminders` helper logically belongs to a lower
layer (utils or a dedicated shared module) than either tool or agent.

### F5 — `warn` — Stray non-TypeScript files in `src/permission/`

`src/permission/PermissionView.kt` and
`src/permission/PermissionViewTest.kt` exist in a TypeScript-only source
tree. They are not built, not tested, and not referenced. They look like
leftovers from a different platform's prototype.

### F6 — `info` — Orphan SQL migration with no SQLite dependency

`src/core/migration/20260511173437_session-metadata.sql` exists but
neither `package.json` nor `src/` contains `better-sqlite3` / `sqlite3`
dependencies, and the constitution states sessions persist to the file
system, "no SQLite". Either this is a future plan that needs an ADR and
a loader, or it is dead code.

### F7 — `info` — `role-architecture.md` module map is stale

The role file lists ~10 modules; reality is 32. New agents reading the
role file will build a mental model that does not match the codebase.
This is informational because the canonical view is `docs/ARCHITECTURE.md`,
which has now been updated.

### F8 — `info` — ESM `.js` import discipline is clean

A naive grep finds 18 hits, but every one of them is inside a JS source
template literal in a test fixture (`src/context/__tests__/ranking.test.ts`),
not a real import. Top-of-line `import` / `export` statements all carry
`.js` extensions on local relative imports.

## Recommendations

1. **Backfill ADRs** for the 17 already-merged top-level directories. A
   single consolidated ADR 002 listing each directory with a one-line
   responsibility + accepted-as-baseline status is preferable to 17
   individual retroactive ADRs.
2. **F2 fix** (`src/cli/commands/models.ts`): move the `DeploymentApi`
   call into `src/providers/` (e.g. `listDeployments()` exported from
   `providers/index.ts`) and have the CLI command consume that. This is
   a small refactor (one helper extraction) and removes a clear
   constitution III violation.
3. **F3 fix** (`src/core/agenticChat.ts`): re-export `CompletionResult`
   and `TokenUsage` from `src/providers/index.ts` and update the import.
   Trivial and improves provider-swap ergonomics.
4. **F4 decision needed**: either move `agentsMdReminders.ts` out of
   `src/agent/` into a lower layer (e.g. `src/agent-md/` or as a
   utility under `src/utils/`), or write an exception ADR formally
   permitting `src/tool/tools/*` to import this specific helper. The
   former is preferred because the helper has no agent-runtime
   dependency — it only walks paths and attaches metadata.
5. **F5 cleanup**: delete `src/permission/PermissionView.kt` and
   `PermissionViewTest.kt`. They cannot be doing useful work in an ESM
   TypeScript build.
6. **F6 decision**: open a tracking issue — keep the SQL file (and add
   ADR + dep + loader) or remove it.
7. **F7**: update `.github/prompts/role-architecture.md` module map to
   match the current `src/` layout, OR (better) point it at
   `docs/ARCHITECTURE.md` as the source of truth so it stops drifting.

## ADR needed?

Yes. This run added **ADR 001 — Establish ADR Process and Layering Rules**
(`docs/adr/001-establish-adr-process.md`). It introduces the directory,
the required sections, the trigger list, the layering rules, and a
follow-up plan for ADRs 002..005 covering the items in F1, F4, F5/F6,
and F2. No feature code was touched.
