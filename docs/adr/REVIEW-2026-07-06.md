### F17 -- `block` -- ADR-001 rule-4 violations in `src/tool/` (previously partially covered by F4)

ADR 001 rule 4 states: `src/tool/ -> may import from bus, permission,
config, utils. NOT from cli, core, agent, providers.`

Current violations (excluding tests):

- `src/tool/tools/edit.ts:9` -> `../../agent/agentsMdReminders.js`
- `src/tool/tools/read.ts:19` -> `../../agent/agentsMdReminders.js`
- `src/tool/tools/multiedit.ts:9` -> `../../agent/agentsMdReminders.js`
- `src/tool/tools/write.ts:11` -> `../../agent/agentsMdReminders.js`
- `src/tool/tools/grep.ts:19` -> `../../agent/agentsMdRemindersForPaths`
- `src/tool/tools/task.ts:7` -> `../../agent/index.js`
  (`getAgentRegistry`, `Agent`)
- `src/tool/tools/task.ts:8` -> `../../core/costTracker.js`
  (`getCostTracker`, `TaskUsageSummary`)
- `src/tool/tools/memory.ts:8-9` -> `../../core/memory.js`
  (`getMemoryManager`, `MemoryEntry`, `MemoryType`)

The agents-md-reminders coupling is the same F4 issue from prior
reviews (five callers, same shared helper). The two `src/core/`
imports from `task.ts` and `memory.ts` are new to the finding list: I
audited them this cycle and they were not enumerated in
`REVIEW-2026-06-29` beyond the aggregate "F4 unchanged" note.

Fix options (all valid, pick one per import group):

1. Move `agentsMdReminders` from `src/agent/` to `src/tool/` (it is a
   pure string-builder, no agent dependencies) and update the five
   callers. Closes the reminder subset of F17 entirely.
2. Move `costTracker` into `src/tool/` or a shared `src/observability/`
   layer that `src/tool/` is explicitly allowed to import from (this
   would require an ADR-002 amendment to rule 4).
3. Move `memory` (currently in `src/core/memory.ts`) into
   `src/tool/tools/memory-store.ts` -- it is exclusively used by the
   memory tool.
4. For `task.ts` importing `getAgentRegistry`, invert the dependency:
   expose the agent registry via `src/bus/` or accept the registry
   as a constructor argument at tool-registration time.

None of these require a new top-level directory. Choose whichever
requires the fewest file moves; document the choice in an ADR only
if it changes rule 4 (option 2), otherwise a plain `refactor(tools)`
PR is enough.

### F18 -- `block` -- ADR-001 rule-4 violation in `src/permission/`

ADR 001 rule 4 states: `src/permission/ -> may import from config,
utils only.`

- `src/permission/index.ts:26` -> `import { getAllToolNames } from
  '../tool/registry.js';`

The permission layer needs to know the set of registered tool names to
validate tool-name globs (introduced with the `AllowEverythingPermission`
work in #903). This is a legitimate architectural coupling that ADR 001
did not anticipate.

Fix: invert the direction. `src/tool/registry.ts` already exports
`getAllToolNames`; have the tool registry *push* the list to a
consumer, or have `src/permission/` accept the list at construction
time from whoever bootstraps both (currently `src/cli/`). Alternatively,
if inversion is impractical, file **ADR 003** amending rule 4 to
permit `permission/ -> tool/registry.ts` specifically (not the whole
`tool/` subtree).

### F19 -- `info` -- new tool-layer growth without an ADR

The `role-architecture.md` module map lists `src/tool/tools/*.ts` as
"built-in tool implementations". `src/tool/tools/` now contains 30+
files including `agent-manager.ts`, `background-process.ts`,
`browser.ts` (13.3 KB), `codesearch.ts` (14.6 KB), `definitions.ts`
(14.5 KB), `diagnostics.ts`, `notebook.ts`, `read-office.ts`,
`recall.ts`, `repo-clone.ts`, plus `background-process/` and `shell/`
sub-directories.

No individual tool is architecturally significant on its own, but the
aggregate has crossed the point where "list them in
`docs/ARCHITECTURE.md`" is impractical. Recommend that ADR 002 include
a rule: the tool registry catalogue is a docs artefact
(`docs/TOOLS.md` or a section of `docs/ARCHITECTURE.md`) that is
regenerated from `src/tool/tools/*.ts` in CI, not hand-maintained. No
`block` here, just a paper-trail request.
