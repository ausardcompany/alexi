## Critical (block)

### C1 -- `xlsx@0.18.5` has Prototype Pollution + ReDoS with no upstream fix

- File: `package.json:60` (direct dependency)
- Advisories:
  - GHSA-4r6h-8v6p-xvw6 (Prototype Pollution in SheetJS)
  - GHSA-5pgg-2g8v-p4x9 (SheetJS ReDoS)
- `fixAvailable: false` per `npm audit --omit=dev --json`.
- Used by: `src/tool/tools/read-office.ts:113` (XLSX-to-CSV preview in
  the `read` tool). Reachable from any agent that reads `.xlsx`/`.xlsm`
  files supplied by the user or an MCP server.
- Recommended remediation:
  1. Move `xlsx` import behind an explicit env flag
     (e.g. `ALEXI_ENABLE_XLSX=true`) and have it deny by default, OR
  2. Replace `xlsx` with `exceljs` (actively maintained, ESM-friendly) or
     a thin in-house ZIP+XML CSV extractor for the small surface we use
     (`readFile`, `utils.sheet_to_csv`).
- Why this is `block`: the `read` tool is invoked by automated agents
  without per-call user approval (`actions: ['read']` matches the
  `default-read-allow` rule in `src/permission/index.ts:679`).
  Adversarial `.xlsx` content can therefore reach the prototype-pollution
  gadget without a human in the loop.

### W3 -- Permission gate: `ExternalDirectoryPermission.evaluate()`
uses a naive `startsWith()` containment check

- File: `src/permission/external-directory.ts:43`
- Issue 1: `normalizedRequest.startsWith(normalizedConfig)` has no path
  separator boundary check. Allowing `/srv/alexi-data` would also
  authorise `/srv/alexi-data-evil/secret`.
- Issue 2: `normalizePath()` (`:21`) lowercases the path. On Linux
  (case-sensitive FS) this destroys case fidelity in the comparison,
  meaning the allowlist for `/Project/Foo` will also accept
  `/project/FOO`. On the runner, where alexi-bot runs, this is a real
  bypass surface -- agents can be tricked into reading
  `/Var/Tmp/...secrets...` via a case-mangled config entry.
- Compare with `src/utils/filesystem.ts:16` (`containsPath`) which gets
  this right: it normalises via `path.resolve` and checks the remainder
  starts with `/` (line 32).
- Recommended fix: replace the body of `ExternalDirectoryPermission.evaluate()`
  with a call to `containsPath(config.path, requestedPath)` (or its
  `safePathCheck` async cousin) instead of reimplementing it. Drop the
  `.toLowerCase()` entirely.
- Severity `warn`, not `block`, because `PermissionManager` (the main
  enforcement path) uses `containsPath` correctly via `isExternalPath()`
  (`src/permission/index.ts:369`). `ExternalDirectoryPermission` is only
  consulted in the cline-style "Ask mode" external directory flow; if
  no caller currently relies on this for trust decisions, the impact is
  limited to that mode.

### W4 -- `release.yml` runs `npm ci` and `npm publish` without
`--ignore-scripts`

- File: `.github/workflows/release.yml:85,91`
- `publish-npm` job exports `NODE_AUTH_TOKEN=${ secrets.NPM_TOKEN }` and
  runs `npm ci` (line 85) without `--ignore-scripts`. Any compromised
  transitive dep's `postinstall` would execute with the npm token in
  scope and could publish a malicious replacement package.
- The same job runs `npm publish --access public` with
  `continue-on-error: true` (line 94), which silently masks publish
  failures -- not directly a security issue, but it can hide a token
  scope/MFA misconfiguration.
- Recommended:
  - Add `--ignore-scripts` to both `npm ci` and `npm publish` in the
    `publish-npm` job. (Keep scripts enabled in the build job; it does
    not hold `NPM_TOKEN`.)
  - Remove `continue-on-error: true` from the publish step.

### W5 -- Documented but unread bypass flag remains in the codebase

- File: `src/flag/flag.ts:33` exports `Flag.dangerouslySkipPermissions()`.
- No call site reads it (`grep dangerouslySkipPermissions src/`
  returns only its declaration). Confirms `docs/security/audit-2026-06-08.md`
  finding I1 -- the flag is dead code.
- Risk: a future PR might wire the flag to a CLI option (kilocode/
  opencode have a `--dangerously-skip-permissions` variant), creating a
  blast-radius escalation without a corresponding ADR.
- Recommended: delete the function and the export. If a bypass is ever
  required, reintroduce it via an explicit ADR plus a permission rule
  override path rather than a global boolean. Tracks AGENTS.md /
  baseline rule that the flag stays opt-in only.

### W6 -- Two `pull_request_target` workflows checkout secret-bearing
context -- currently safe, audit-worthy

- Files: `.github/workflows/on-release-merge.yml:4`,
  `.github/workflows/update-homebrew.yml:9`
- Both pin `ref: master` (not `head.sha`) on checkout. Good.
- Both gate on `pull_request.merged == true` and a `head.ref` prefix
  filter (`release/*`, `auto/sync-upstream-*`). Good.
- `on-release-merge.yml:27-29` derives `VERSION=${BRANCH#release/}` from
  `github.event.pull_request.head.ref` and passes it through
  `sed 's|TAG_PLACEHOLDER|${{ steps.version.outputs.tag }}|'` (line 103).
  Branch names cannot include `|`, but injecting `;` or backticks is
  possible. The `startsWith('release/')` filter plus PR-merge gate makes
  exploitation require admin merge approval. Treat as `warn` only because
  the surface is non-zero, not because it is currently exploitable.
- Recommended hardening: pass `head.ref` into the env block instead of
  via expression substitution in `run:`, then read it as `$VERSION` --
  same pattern `agent-factory.yml:144-150` already uses for
  `RAW_TASK_PROMPT`.

## Info

### I1 -- `kilo-output.log` retained for 30 days as artefact

- File: `.github/workflows/agent-factory.yml:265`
- The agent's stdout/stderr is captured to `kilo-output.log` and uploaded
  as `factory-<role>-<run-id>` for 30 days. The same log's last 80 lines
  are also echoed to step logs (line 226). GitHub masks single-line
  secret values automatically, but multi-line `AICORE_SERVICE_KEY` JSON
  can escape masking when re-emitted by tools.
- Recommended: scrub `AICORE_SERVICE_KEY` from logs explicitly before
  emitting (`grep -v -F "$AICORE_SERVICE_KEY"` filter, or unset the env
  var after `kilo run` returns).

## Recommendations

Priority order (lowest blast radius first, biggest risk reduction):

1. **Apply `npm audit fix` (no `--force`)** on a dedicated PR. Validate
   `marked@15.x` and `marked-terminal@7.x` are preserved in
   `package-lock.json`. Run the full CI gate locally
   (`lint -> typecheck -> format:check -> test:coverage -> build`).
   Closes W1 plus partial W2, I3, I4. Estimated diff: lockfile-only.
2. **Fix `ExternalDirectoryPermission`** (W3) -- replace the body of
   `evaluate()` with `containsPath(config.path, requestedPath)`. Add
   a regression test in `tests/permission*` for:
   - Prefix bypass: allowlist `/a` should NOT match `/abc`.
   - Case sensitivity: on Linux, allowlist `/A` should NOT match `/a`.
3. **Harden `release.yml`** (W4) -- add `--ignore-scripts` to
   `npm ci` and `npm publish` in the `publish-npm` job, remove
   `continue-on-error: true` from the publish step.
4. **Delete unreachable `Flag.dangerouslySkipPermissions()`** (W5).
   Re-link to the audit-2026-06-08 I1 entry in the PR description.
5. **Replace `xlsx` or gate it behind an env flag** (C1). Track as a
   `security` label issue; this is non-mechanical and needs design
   input from `role-architecture` (changing the `read` tool's office
   document API surface).
6. **Harden `on-release-merge.yml`** (W6) -- move `head.ref` through an
   env block before any shell substitution. Track as `ci(ci)`.
